# 2026-05-22 K8S 클러스터 VM 재시작 후 복구 (9일차)

## 증상
VM 전체 재시작 후 파드들이 `ContainerCreating` 상태에서 멈춤

## 원인 분석

### 1. SSH 접속 불가
- `authorized_keys` 파일 자체가 없었음 (`/home/junhyung2/.ssh/` 디렉토리 미존재)
- kubeconfig도 없었음 (`~/.kube/` 디렉토리 미존재)
- **조치**: Hyper-V 콘솔에서 직접 로그인 후 수동 생성

```bash
# SSH 키 등록
mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHjDNckCS88Mq1hTK6penVRo1bqSGy9cA64Usfpzrd9A claude-code-agent" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

# kubeconfig 설정
mkdir -p ~/.kube
sudo cp /etc/kubernetes/admin.conf ~/.kube/config
sudo chown $(id -u):$(id -g) ~/.kube/config
```

### 2. Calico CNI Unauthorized 오류
- **에러**: `plugin type="calico" failed (add): error getting ClusterInformation: connection is unauthorized: Unauthorized`
- **원인**: VM 재시작 후 calico-node의 서비스 어카운트 토큰이 만료/무효화
- **조치**: calico-node 데몬셋 재시작

```bash
kubectl rollout restart daemonset calico-node -n calico-system
```

### 3. Istio 파드 자동 복구
- Calico 정상화 후 `ContainerCreating` 상태였던 Istio 파드들이 자동으로 Running 전환
- 별도 조치 불필요

## VM 재시작 시 체크리스트

| 순서 | 확인 항목 | 명령어 |
|---|---|---|
| 1 | Calico 상태 | `kubectl get pods -n calico-system` |
| 2 | Calico 오류 시 재시작 | `kubectl rollout restart daemonset calico-node -n calico-system` |
| 3 | Istio 상태 | `kubectl get pods -n istio-system` |
| 4 | 전체 이상 파드 확인 | `kubectl get pods -A \| grep -v Running \| grep -v Completed` |
